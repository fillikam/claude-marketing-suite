# Архитектура Claude Marketing Suite

## Обзор

Claude Marketing Suite построен как набор автономных, но взаимосвязанных скиллов. Каждый скилл решает одну задачу и создаёт один выходной файл. Мета-оркестратор (`/aim report`) может запускать все скиллы последовательно.

---

## YAML-триггеры (.skill файлы)

Каждый скилл имеет `.skill` файл с YAML-конфигурацией:

```yaml
name: aim-audit
description: Полный маркетинговый аудит сайта
triggers:
  - /aim audit
output: MARKETING-AUDIT.md
```

Поля конфигурации:
- `name` -- уникальный идентификатор скилла
- `description` -- краткое описание для каталога скиллов
- `triggers` -- список слэш-команд, активирующих скилл
- `output` -- имя выходного файла

---

## Структурированный вывод

Все скиллы генерируют Markdown-файлы с единообразной структурой:

```markdown
# Заголовок отчёта
> Мета: дата, URL, версия

## Раздел 1
### Подраздел
- Данные в таблицах и списках

## Выводы и рекомендации
- Приоритезированные действия
```

Это обеспечивает:
- Читаемость для людей
- Парсируемость для других скиллов
- Совместимость с любым Markdown-редактором

---

## Guardrails (ограждения качества)

Каждый SKILL.md содержит встроенные guardrails:

1. **Минимальная глубина анализа** -- скилл не может выдать поверхностный ответ
2. **Обязательные разделы** -- каждый выходной файл должен содержать определённые секции
3. **Anti-fluff** -- запрет на общие фразы типа «важно помнить» без конкретики
4. **Числовые метрики** -- где возможно, скиллы дают скоринг и оценки
5. **Actionable-рекомендации** -- каждый совет должен быть конкретным действием

---

## Fallback-протокол

Когда скиллу недостаточно данных для полного анализа:

1. **Уровень 1**: Скилл явно указывает, какие данные отсутствуют
2. **Уровень 2**: Использует отраслевые бенчмарки вместо конкретных данных
3. **Уровень 3**: Помечает раздел как `[ТРЕБУЕТ ДАННЫХ]` с инструкцией, что предоставить
4. **Никогда**: Скилл не выдумывает данные и не делает необоснованных утверждений

## Кросс-скилл интеграция

### Механизм

При запуске каждый скилл сканирует текущую директорию на наличие выходных файлов других скиллов:

```
Текущая директория
├── MARKETING-AUDIT.md      ← данные для всех скиллов
├── LANDING-CRO.md           ← данные о конверсиях
├── COMPETITOR-REPORT.md     ← конкурентный контекст
├── BRAND-VOICE.md           ← тональность для контента
├── AUDIENCE-RESEARCH.md     ← портрет ЦА
├── EMAIL-SEQUENCES.md       ← email-стратегия
├── SOCIAL-CALENDAR.md       ← контент-план
├── AD-CAMPAIGNS.md          ← рекламные кампании
└── HEADLINES.md             ← заголовки
```

### Граф зависимостей

```
audit, landing, competitors (независимые, базовый слой)
        ↓
brand, audience (используют данные базового слоя)
        ↓
emails, social, ads, headlines (используют все предыдущие)
        ↓
report (собирает всё в единый документ)
```

### Автономность

Каждый скилл может работать без зависимостей. Интеграция улучшает качество, но не является обязательной.

---

## Безопасность

- Скиллы не имеют доступа к API-ключам или секретам
- Все данные обрабатываются локально через Claude Code
- Выходные файлы создаются только в текущей директории
- Никакие данные не отправляются на сторонние серверы (кроме стандартного API Claude)
